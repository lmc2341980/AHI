# Purpose

This document defines the standard process for meetings, discussions, decisions and GitHub implementation within the Artificial Hybrid Intelligence (AHI) ecosystem.

Every important discussion shall leave reusable knowledge inside GitHub.

GitHub is the Source of Truth.

---

# Meeting Workflow

Every meeting shall follow the same lifecycle.

Question

↓

Discussion

↓

Proposal

↓

Decision

↓

GitHub Asset

↓

Approval

↓

DONE

↓

GitHub Commit

↓

Document Registry Update

---

# Standard Meeting Roles

## Founder

Provides the final approval.

---

## PM (Project Manager)

Defines objectives.

Prioritizes work.

Approves project execution.

---

## P (Professor)

Provides professional knowledge.

Analyzes risks.

Proposes architectures.

---

## L (Team Leader)

Reviews impacts within the responsible Repository or Folder.

Ensures implementation quality.

---

## S (Secretary)

Records discussions.

Creates Meeting Minutes.

Updates standards.

Maintains documentation.

Updates the Document Registry.

---

## A (Assistant)

Collects opinions.

Finds conflicts.

Produces implementation plans.

Generates GitHub assets.

Suggests next actions.

The Assistant never makes final decisions.

---

# Meeting Output

Every meeting should produce one or more of the following outputs.

- Meeting Minutes
- Draft
- Decision
- Standard
- Architecture
- Implementation
- Code
- Product

Every meeting should leave reusable knowledge.

# Decision Rules

A decision becomes official only after Founder approval.

Decision lifecycle:

Proposal

↓

Discussion

↓

Founder Approval

↓

GitHub Update

↓

DONE

↓

Registry Update

---

# DONE Rule

The word

DONE

means that:

- The GitHub file has been created or updated.
- The Commit has been completed.
- The content is accepted.
- The task may continue.

Without DONE, work remains in Draft status.

---

# Meeting Minutes

Every important meeting shall generate a Meeting Minutes document.

The Meeting Minutes should include:

- Meeting ID
- Date
- Participants
- Purpose
- Main discussion topics
- Decisions
- GitHub assets created
- Next actions

Meeting Minutes preserve project history.

---

# Decision Documents

Each important decision should receive a unique identifier.

Example:

DECISION-20260705-001

Decision documents should include:

- Background
- Decision
- Reason
- Expected impact
- Related documents

---

# Question–Answer Principle

Every question should lead toward a decision.

Every answer should generate the next meaningful question.

Meetings are evolutionary processes rather than isolated conversations.

---

# GitHub Asset Requirement

Whenever implementation is requested, the Assistant shall always generate:

- Repository
- Folder
- File
- Document ID
- Commit Message
- Complete Markdown

No task is considered complete until these assets exist.

---

# Continuous Improvement

Meeting standards may evolve.

Improvements shall be proposed through Draft documents and approved following the standard decision process.

---

# Guiding Principle

Meetings create ideas.

Decisions create direction.

GitHub preserves knowledge.

AHI continuously evolves through structured collaboration.
